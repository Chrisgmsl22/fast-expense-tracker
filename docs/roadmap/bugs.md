# Bug register

Tracks bugs found by **using the app in prod** — defects that don't belong to a
phase's planned slices. The source of truth is [`bugs.json`](./bugs.json); the
table below is generated from it by `pnpm bugs:status --write` (do not hand-edit
the block). Feature-requests are **not** here — those go in the roadmap slices.

**Each bug is its own mini-slice:** one `fix/<id>-…` branch, one PR, the normal
[slice lifecycle](../conventions/slice-planning.md). Work one at a time.

**Fields** (per entry in `bugs.json`): `id` (`BUG-n`), `title`, `description`,
`proposedSolution`, `status`, `spec` (path when one is written), `pr` (link once
open), `issue` (GitHub issue number it was drained from, else null).

**Status flag:** `open → in-progress → fixed → merged`.

**Workflow**

1. Log the bug in `bugs.json` (`status: "open"`).
2. Pick one up: branch off up-to-date `main`, set `status: "in-progress"`.
3. Fix, test, review, PR — link the `pr`; set `status: "fixed"` on approval.
4. On merge: set `status: "merged"`. Run `pnpm bugs:status --write` to refresh this table.

**GitHub Issues = capture inbox.** Report most bugs in a Claude session (straight
into `bugs.json`). When away, file a GitHub issue instead; next session, **drain**
it: pull open issues (GitHub MCP) → for each, check `bugs.json`. If it's new, add
an entry with `issue: <n>`. If it's already tracked, link the `issue` field and
close the GitHub issue with a "tracked as BUG-n" note. No background automation —
matching + wording stay human-in-the-loop, run on request.
