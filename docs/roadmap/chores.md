# Chore register

Tracks **feedback-driven features and chores** — work that comes from using the
app, not from the planned roadmap. This keeps [`slices.json`](./slices.json) the
pristine phased trajectory: feedback work lives here, never as drifting `2.13,
2.14…` slices. Defects go in the [bug register](./bugs.md) instead; both share
the same engine (`scripts/lib/register-status.ts`).

The source of truth is [`chores.json`](./chores.json); the table below is
generated by `pnpm chores:status --write` (do not hand-edit the block).

> **Money-model rework (CHORE-4 → CHORE-3 → CHORE-5):** the full design, decisions,
> code-impact, prod-migration cautions, and a cold-start kickoff checklist live in
> [`../specs/0005-cash-basis-money-model.md`](../specs/0005-cash-basis-money-model.md).
> Start with **CHORE-4** (write its ADR first, for sign-off).

**Each chore is its own mini-slice:** one `feat/<id>-…` branch, one PR, the normal
[slice lifecycle](../conventions/slice-planning.md). Work one at a time.

**Fields** (per entry in `chores.json`): `id` (`CHORE-n`), `title`, `description`,
`plan` (approach; may point to a `spec`), `status`, `spec`, `pr`, `issue`.

**Status flag:** `open → in-progress → shipped → merged`.

**Workflow**

1. Log the chore in `chores.json` (`status: "open"`).
2. Pick one up: branch off up-to-date `main`, set `status: "in-progress"`.
3. Build, test, review, PR — link the `pr`; set `status: "shipped"` on approval.
4. On merge: set `status: "merged"`. Run `pnpm chores:status --write` to refresh this table.
